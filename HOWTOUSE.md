# How to use this repo

The author's manual. If `HOSTING.md` is the operator's manual (how deploys,
DNS, and certs work), this is the editor's manual (how to change what the site
actually says). Read this once; you'll rarely need the rest.

---

## The mental model (the single most important thing)

This site is a **static site**. That means:

1. You edit **source files** on your computer (`.astro`, `.mdx`, `.ts`, `.css`).
2. A tool called **Astro** reads those source files and **generates** plain
   `.html`, `.css`, and `.js` into a folder called `dist/`.
3. Those generated files get uploaded to GitHub Pages, which serves them to
   visitors.

You never edit the generated HTML. Every time you build, `dist/` is rewritten
from scratch from the sources. If you open `dist/cv/index.html` and tweak it,
your change vanishes the next time you run `npm run build`.

Why do it this way instead of writing HTML directly? Two reasons:

- **Content and structure stay separate.** Your CV is a list of jobs in
  `src/data/cv.ts`. The template that renders jobs into HTML is in
  `src/pages/cv.astro`. Change a job — the content file. Change how jobs look —
  the template file. You don't have to copy-paste the same HTML for every
  entry.
- **Validation.** The build step (`npm run build`) checks that every essay has
  the required frontmatter, that links aren't broken to non-existent pages, and
  that TypeScript types line up. Plain HTML gives you no safety net.

A static site is just a **build pipeline**: sources go in, HTML comes out. The
build runs locally when you type `npm run build`, and it runs automatically in
GitHub Actions every time you push to `main`.

## Tour of the repo

Everything you'll edit lives in `src/`. Here's what each folder is for:

```
src/
├── content/
│   ├── config.ts             schema for essay frontmatter (Zod)
│   └── essays/               *.mdx — one file per essay
│       └── hello-world.mdx

├── data/
│   ├── cv.ts                 CV content: profile, education, work
│   └── appearances.ts        (unused right now — scaffolding for talks/podcasts)
│
├── components/               small reusable pieces
│   ├── Header.astro          site header (wordmark + nav)
│   ├── Footer.astro          site footer (links + copyright)
│   └── EssayCard.astro       essay preview used on home + archive
│
├── layouts/                  page shells
│   ├── BaseLayout.astro      every page uses this (head, header, footer, fonts)
│   └── EssayLayout.astro     wraps a single essay
│
├── pages/                    one file = one URL
│   ├── index.astro           → /              (homepage)
│   ├── archive.astro         → /archive
│   ├── cv.astro              → /cv
│   ├── colophon.astro        → /colophon
│   └── essays/[slug].astro   → /essays/<slug> (dynamic route for each essay)
│
└── styles/
    └── global.css            all styles live here. tokens at the top.

public/                       files copied verbatim to dist/
├── favicon.svg
└── CNAME                     tells GitHub Pages which domain to serve
```

**What's not in `src/` but matters:**

- `astro.config.mjs` — tells Astro what site URL we're building for, and to
  enable MDX.
- `package.json` — lists dependencies. `npm install` reads this.
- `.github/workflows/deploy.yml` — the GitHub Actions workflow that builds and
  publishes on every push to `main`.
- `HOSTING.md` — operator's manual (DNS, certs, rollback).
- `HOWTOUSE.md` — this file.
- `README.md` — high-level overview.

**What you never edit:**

- `dist/` — build output. Regenerated on every build. Don't commit it
  (`.gitignore` already excludes it).
- `node_modules/` — installed dependencies. Regenerated by `npm install`.
  Don't commit it.
- `.astro/` — Astro's internal cache. Don't commit it.

## The core loop: edit → preview → ship

Every change, no matter how small, follows the same four steps.

### 1. Pull (only if needed)

Only if you've edited from another machine or from github.com in a browser:

```bash
cd C:\Users\Gamer\Projects\alfredbirkelund.com
git pull
```

If everything is on this one machine, skip it. (`git pull` fetches any commits
made elsewhere and merges them into your local copy, so you don't push on top
of a stale branch.)

### 2. Edit in `src/`

Open whichever file you want to change in a text editor. Save when done.

### 3. Preview locally

```bash
npm run dev
```

This starts Astro's dev server at `http://localhost:4321` with **live reload**
— the page updates in your browser every time you save a file. Dev mode is
forgiving: it doesn't run the full type check, so you can iterate fast.

Stop the dev server with `Ctrl+C` when you're done.

Optionally, before pushing, run the full build to catch anything dev mode
skipped:

```bash
npm run build
```

This runs `astro check` (typechecking, frontmatter validation) *and* `astro
build` (generates `dist/`). If it passes locally, it'll pass in CI.

### 4. Ship

```bash
git add .
git commit -m "short message — e.g. Update CV"
git push
```

That's it. The GitHub Actions workflow sees the push, runs the build on
their servers, and uploads `dist/` to Pages. Live at
`https://alfredbirkelund.com` within about a minute.

You can watch the deploy in real time at:
`https://github.com/Lorenz0n/alfredbirkelund/actions`

Or from the terminal: `gh run watch`.

## What each git command actually does

Because you asked to level up — these five commands cover 95% of what you'll
ever need.

| Command | What it does |
| ------- | ------------ |
| `git status` | Shows which files have been changed since the last commit. Your first move when you don't remember what you edited. |
| `git diff` | Shows the actual line-by-line changes in those files. |
| `git add <file>` | Stages a file — marks it as "include in the next commit". `git add .` stages every changed file. |
| `git commit -m "..."` | Creates a snapshot of the staged files. The message is a note-to-future-self explaining *what* and *why*. Immutable once created. |
| `git push` | Uploads your new commits to GitHub. Only after this does the deploy run. |

A **commit** is a permanent labeled snapshot. A **push** sends commits from
your machine to GitHub. You can commit many times locally before pushing —
useful if you want to checkpoint work mid-change without triggering a deploy.

### Commit message conventions

Short, imperative, one line. Describe the change, not the mechanics.

Good:

- `Update CV: add RA role at SDU`
- `Add essay: the case against predicting rate cuts`
- `Fix mobile padding bug`
- `Tighten footer spacing on narrow screens`

Less good:

- `stuff` — future you will hate present you.
- `wip` — fine for local checkpoints, don't push these.
- `Updated the src/data/cv.ts file to add a new entry` — too long, too focused
  on the *what* instead of the *why*.

## Common edits, with recipes

### Publishing a new essay

Essays live in `src/content/essays/` as `.mdx` files. MDX is Markdown that
also lets you drop in components — but for plain prose you only need Markdown.

> For everything beyond plain prose — images, footnotes, headings, series,
> collapsible explainers, pullquotes — see **`WRITING.md`**. Two living
> reference essays sit in `src/content/essays/`: `test-headings.mdx` (single
> page with TOC) and `test-series/` (multi-page with sidebar).

1. Create `src/content/essays/<slug>.mdx`. The filename (minus `.mdx`)
   becomes the URL: `rate-cuts.mdx` → `https://alfredbirkelund.com/essays/rate-cuts/`.
2. Start with the required frontmatter — this is the YAML block at the top
   between `---` fences:

   ```mdx
   ---
   title: "The case against predicting rate cuts"
   subtitle: "Forecasters keep doing this, and keep being wrong."
   date: 2026-06-14
   category: economics
   favorite: false
   draft: false
   ---

   Your prose starts here. **Bold**, *italic*, [links](https://example.com),
   and ## headings all work.
   ```

3. Fields:

   | field      | required | notes                                                                 |
   | ---------- | -------- | --------------------------------------------------------------------- |
   | `title`    | yes      | the essay's title                                                     |
   | `subtitle` | no       | shown in italics under the title; omit the line if you don't want one |
   | `date`     | yes      | `YYYY-MM-DD`                                                          |
   | `category` | yes      | one of `economics`, `finance`, `technology`, `philosophy`, `misc`     |
   | `draft`    | no       | `true` hides it from the site entirely; defaults to `false`           |
   | `favorite` | no       | `true` surfaces it on the home "Favorites" section                    |

4. Preview with `npm run dev`, visit the URL. Ship with the usual git flow.

**Drafts are your friend.** Push work-in-progress freely with `draft: true` —
they won't appear on the site. Flip to `false` when ready to publish.

### Updating the CV

All CV content lives in `src/data/cv.ts`. You edit TypeScript object literals,
but you're just changing strings — no programming required.

To add a new work entry:

```ts
export const work: Work[] = [
  { /* existing entry... */ },
  {
    role: 'Research Assistant',
    organization: 'SDU Economics Department',
    location: 'Odense, Denmark',    // optional — delete the line if N/A
    dates: '2025–present',
    description: 'What you did, in one or two sentences.',
    // url: 'https://...',          // optional — delete if N/A
  },
];
```

To remove an entry, delete the whole `{ ... }` block including the trailing
comma. Order in the array = order shown on the page (top to bottom).

Education works the same way — the `education` array in the same file.

Profile block (name, tagline, email, location, social links) is the `profile`
object at the top.

The template at `src/pages/cv.astro` loops over these arrays and renders them.
You only need to touch `cv.astro` if you want to change **structure** — e.g.
add a new section like "Publications" or reorder.

### Editing the homepage intro or the About paragraph

Both live in `src/pages/index.astro` as plain HTML-ish text. Find the
relevant `<p>` or `<section>` tag, change the words, save.

### Tweaking styles

All CSS is in `src/styles/global.css`. Near the top you'll find "design
tokens" — CSS custom properties (`--color-bg`, `--size-body`, etc.) that the
rest of the file uses. Change a token to re-skin everything at once; change a
specific selector (`.essay__title { ... }`) to adjust one thing.

Mobile-specific tweaks go inside the `@media (max-width: 640px)` block at the
bottom.

## Good-to-knows

### The build is your safety net

`npm run build` catches:

- missing or misspelled frontmatter fields in essays
- TypeScript errors (e.g. a typo in `cv.ts`)
- invalid internal links
- syntax errors in `.astro` files

If the build passes locally, it'll pass in CI. If CI is green, the site is
updated. Trust the build.

### Drafts let you push freely

`draft: true` in an essay's frontmatter excludes it from every build. You can
commit and push 50% of an essay with `draft: true` and nothing appears on the
live site. Flip to `false` to publish.

### The live site keeps working while a broken build sits in CI

If you push something that breaks the build, GitHub Actions shows a red X and
emails you — **but the previous deploy keeps serving**. Visitors see no
downtime, no error page. You fix it, push again, and the site jumps to the
new version.

### Rolling back

If you ship a version you regret:

```bash
git revert HEAD       # creates a new commit that undoes the last one
git push
```

Or: go to the **Actions** tab on GitHub, find the last good run, click
"Re-run all jobs". The older artifact gets redeployed.

Never use `git reset --hard` or force-push on a shared branch — those rewrite
history and can lose work.

### Editing from GitHub's web UI (useful on mobile)

Click any file on github.com → pencil icon → edit → "Commit changes". Same
auto-deploy runs. Good for typo fixes when you're not at your computer. **When
you get back to your computer, `git pull` first** — otherwise your next local
commit will be based on stale state.

### What *not* to commit

- `dist/`, `node_modules/`, `.astro/` — all in `.gitignore`, stay out.
- `.env` files with secrets — also gitignored. This site doesn't use any, but
  if you ever add one, don't commit it.

### What if `npm install` / `npm run dev` fails?

```bash
# nuclear option — wipes installed packages and reinstalls from scratch
rm -rf node_modules package-lock.json
npm install
```

This is safe. `package.json` is the source of truth; `node_modules` and
`package-lock.json` are both regenerable.

## When things go wrong

### "I pushed and the site didn't update"

1. Check the Actions tab — was there a red X? Read the failing step.
2. If green, but the site still shows old content, your browser or ISP is
   caching. Try incognito / hard refresh / a different network.
3. If it's been more than 2 minutes and CI is green and incognito also shows
   old content: file a ghost in the machine issue and go touch grass. (Rare.)

### "astro check fails on a new essay"

Almost always a frontmatter issue:

- `category` must be one of the allowed values (see the table above).
- `date` must be in `YYYY-MM-DD` format.
- `title` is required.

The error message points to the file and usually the field.

### "the mobile layout broke"

`src/styles/global.css` is the only place that controls styling. Mobile rules
are inside `@media (max-width: 640px) { ... }`. Desktop rules are outside it.
If something broke, revert the last commit touching `global.css`:

```bash
git log --oneline src/styles/global.css     # find the offending commit
git revert <sha>
git push
```

### "I accidentally deleted something and pushed it"

Git never forgets. Find the commit where the thing still existed:

```bash
git log --oneline
git show <sha>:src/path/to/file > restored.txt
```

Or revert the deleting commit entirely with `git revert <sha>`.

## Where to go from here

If you want to grow the site beyond what's here now — add a newsletter, a tag
index, image support for essays, a search — the starting point is the Astro
docs: https://docs.astro.build. The framework is small; you can read the
"Pages", "Layouts", and "Content Collections" sections end-to-end in an
afternoon and have a mental model of everything this site does.

If you want to swap Pages for a fancier host (Vercel, Netlify, Cloudflare
Pages), the whole site is portable — they all accept a static `dist/` folder
via `npm run build`. See `HOSTING.md` for why we picked GitHub Pages.
